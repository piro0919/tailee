# CLAUDE.md

Tailee のリポジトリで Claude Code が作業する際のガイドラインです。

## Tailee とは

**ペットとの「なんでもない日常」が勝手に貯まり、勝手に日記がつき、アルバムになるサービス。**

犬がご飯を食べている姿は、意外と誰も撮らない。撮ろうと思わないから撮らない。そこを残す。
一日ぶんは何でもないが、一年ぶん積もると意味を持つ。ペットの夕飯はおよそ一万回で終わる、という有限性がこの製品の芯にある。

**「そこを残す」であって「そこだけを残す」ではない。** 見せられる絵となんでもない絵は、
普通のアルバムなら両方入っている。SNS 向けに撮りに行く動きが出ても問題ない
（2026-07-17 ユーザー「アルバムなんて、そこらへん両方入ってるのが普通じゃない？」）。
純度を守る方向に設計を寄せないこと。

### 決まっていること

| 項目           | 内容                                                                     |
| -------------- | ------------------------------------------------------------------------ |
| **主体**       | アルバム。見守りカメラではない                                           |
| **見る人**     | 基本は飼い主本人。ただし共有を否定しない（下2行）                        |
| **ライブ配信** | 見守りカメラの映像を、そのままライブ配信したい                           |
| **SNS**        | アプリ内に、共有のインスタのようなものを作りたい                         |
| **カメラ**     | 古いスマホを想定。ただし使うかはユーザー次第。専用ハードは作らない       |
| **置き方**     | 餌皿の横などに置きっぱなし。毎日同じ場所で起きる数分を狙う               |
| **家にいる時** | ウェイクワードで録画・録音を開始。餌皿を持っていて画面を触れないから声   |
| **手持ち撮影** | 画面のボタンで開始。突発的ないい場面はこちら。同じアルバムに入る         |
| **留守中**     | 見守りカメラとして動く。リアルタイム配信であって、録画も録音もしない     |
| **日記**       | 音声から下書きを自動生成。**人も書ける**。実況させないのは入力の話だけ   |
| **展開**       | 海外展開が前提。ひらがな・日本語依存の名前は不可                         |
| **立ち位置**   | 並ぶ棚はペットカメラ（Furbo等）ではなく、子供の成長アルバム（みてね等）  |

### 設計上の分かれ目（重要）

**日記は「実況させない」。** カメラに向かって「今ご飯を食べています」と説明させる設計にすると三日で終わる。
飼い主はもともとペットに話しかけている（「はい、ごはん」「昨日は残したのに」）。
**その独り言を拾って、機械の側が文章を立ち上げる。** 人間に余計な発話を一言もさせない。

映像から分かるのは「11時に日向で寝ていた」まで。
「今日ワクチンを打った」「はじめて他の犬と挨拶できた」は飼い主の頭の中にしかない。
声を**指示ではなく素材として**拾えば、そこも残る。

**ただし日記を製品の核として扱わないこと。** 2026-07-17 にユーザーは
「日記はそこまでなんだよな、あくまでアルバムが主体という感じ」と述べている。
日記はアルバムに付くものであって、日記のために作っているのではない。

### 日記は人も書く（2026-07-17 決定）

**「実況させない」は入力の話であって、「書かせない」ではない。** 混同しないこと。一度やった。

機械が音声から下書きを立ち上げる。その上で、飼い主がその日のペットの様子や
プライベートなことを書き足せる。書いたものは最終的にアルバムに反映される。

エディタは **BlockNote**（tuzuri と同じ）。AI の入口は**2つとも入れる**。

| | 何 | tuzuri のどこ |
| --- | --- | --- |
| **A** | エディタ内のインライン AI。選択して書き換え、差分で Accept / Reject | `@blocknote/xl-ai`。丸投げで自前コードほぼゼロ |
| **B** | 左サイドバーのチャット。本文を渡して相談し、提案を Insert で挿入 | 自前。`ChatSidebarContent` + `/api/chat` |

**A を入れる理由**（ユーザー談）: 「日記を書くことに抵抗が強い人もいるだろうし」。
空白のページより、直す方が入りやすい。

**A の値札**（承知の上で入れる）:

- **Yjs が付いてくる。** xl-ai が内部で awareness を使うため、単独ユーザーでも Yjs provider と
  ダミーの awareness が要る。tuzuri は `@ts-expect-error` でねじ込んでいる。Liveblocks は不要
- **AI SDK が2バージョン入る。** xl-ai が v4 系を要求するので、v5 とエイリアスで併存させる

### tuzuri から持ってこないもの

- **Liveblocks 一式** — 共有作品専用。単独ユーザーの日記に不要
- **Handlebars のプロンプトエンジン + テンプレート4テーブル + admin 管理画面** —
  非エンジニアがプロンプトを編集するための仕組み。日記ならコードに2〜3種直書きで足りる
- **スレッド階層**（親子スレッドのマージ）— 長編執筆の段階ワークフロー用。日記は1本でいい
- **クレジット制一式** — 課金しないなら不要。ただし暴走防止の日次上限は別途要る
- **「✍️」絵文字マーカーで Insert ボタンを生やす仕組み** — LLM が絵文字を出し忘れると壊れる。
  複数行の blockquote で挿入順が逆になるバグもある。`streamObject` で
  `{ draft, comment }` を返させる形に作り直す
- **Yjs の状態を `Array.from()` で数値配列 JSON にして jsonb 保存** — 数KB のバイナリが
  数十KB の JSON になり、2.5秒ごとに全量 UPDATE される。**絶対に真似しない**

## 名前

**サービス名は Tailee（テイリー）。** tail に由来する造語。2026-07-17 に決定。

検索した限り同名のブランド・アプリ・商標は無い。日本語の一般語ともぶつからない。
※ これは検索であって商標調査ではない。本気で押さえる段では別途必要。

### 未決

- **アルバムの単位名。** `Taileaf` が候補。leaf は一冊ではなく一枚を指すので、
  「一日ぶん＝一枚の Taileaf、積もって一冊」が意味的には正しい。
  ただし Tailee と Taileaf は /f/ 一音しか違わず、/f/ は最も落ちやすい音。
  声に出すと混同する（「Tailee でしたっけ、Taileaf でしたっけ」）。実画面に並べてから決める。
- **アルバムの提供の仕方。** まだ詰めきれていない。「年一とかかなー」という程度で、
  2026-07-17 時点で不明瞭。ここは決まっていないものとして扱う。
- **ウェイクワード。** 後からカスタマイズできる想定で、名前とは分けて考える（Echo と Alexa の関係）。
  条件は「日常語でない」「同音語がない」「四拍以上」、加えて **Tailee と音が衝突しない**こと。

  **造語は使えない**（Vosk は語彙ベースで、モデルの語彙にない語は認識しない）。実在語から選ぶ。
  仮候補は「あいことば」「かんらんしゃ」「まほうびん」で `src/lib/wake-words.ts` に置いてある。

1. **置きっぱなし端末の待ち受けが持つか**
   常時マイクを開いて音声認識を回し続ける処理は、iOS Safari では画面を閉じると止まる可能性が高い。
   Android でも継続的な集音は削られる。ここが崩れるとウェイクワードという前提ごと崩れる。

3. **電源の本数。** 端末代はゼロでも、置く場所ごとにコンセントとケーブルが要る。

### 台数について

追加台数の原価がゼロなのは、ハードを売る競合（Furbo 等）が絶対に真似できない構造的な強み。
**複数台用意できるなら、それに越したことはない。** 台数が増えれば画角が生まれ、その分アルバムは良くなる。

## 技術構成

- Next.js 16 (App Router) + React 19 + TypeScript + Tailwind CSS v4
- pnpm（`allowBuilds` は `pnpm-workspace.yaml` に書く。pnpm 11 は build script を承認制で止める）
- 単一アプリ構成。モノレポではない
- **Lint は Biome。** ESLint ではない（2026-07-17 決定。個人開発の範囲なら十分、という判断）
- **Supabase** — DB・認証・画像。参考リポジトリ3つとも Supabase で、書き慣れているのが理由
- **ウェイクワードは Vosk**（`use-ear` 経由、2026-07-17 決定）
- **文字起こしは Web Speech API**（2026-07-17 決定）。AmiVoice は使わない
- **日記生成の LLM は OpenAI**（2026-07-17 決定）。AI SDK 経由。
  参考リポジトリ3つとも OpenAI で、nagara の golden test（LLM 出力をキャッシュして
  CI で無料・決定的に回す仕組み）がそのまま効くのが理由。コストは論点ではない
  （1日1回・短い日記なら、モデルを変えても年に数百円の差）

### 音声は二段に分ける

| 何を | 何で | なぜ |
| ---- | ---- | ---- |
| **常時待ち受け**（ウェイクワード） | **Vosk**（オンデバイス） | Web Speech だと earcon が鳴り続ける |
| **録画中の数分**（文字起こし） | **Web Speech API** | 無料。鳴っても1回 |

nagara も分けているが、あちらは書き起こしが **AmiVoice**（従量課金）。**Tailee では使わない。**
毎日・全ユーザーぶんの数分が課金され、動画や配信と違って**逃がす先が無い**ため。
nagara が金を払っているのは介護記録に精度が要るからで、犬に話しかけた独り言に同じ精度は要らない。
「昨日は残したのに」が多少崩れても日記としては成立する。

足りなければ AmiVoice に上げる。逆順（先に AmiVoice）は無駄になる。

### なぜ待ち受けは Web Speech ではなく Vosk か

参考リポジトリが割れている（nagara は Vosk、chappie は Web Speech）が、**条件が違う**。

nagara は Web Speech から Vosk に乗り換えた側で、理由がそのまま Tailee に効く。

> Web Speech API はセッション再起動のたびに OS の認識開始/終了音（earcon）を鳴らし、
> 待受け中ずっとピコピコ鳴り続けるため

餌皿の横に24時間置く端末なので、鳴り続けたら終わる。chappie が Web Speech で平気なのは
店頭のキオスクで、人が来た時だけ喋るから。置きっぱなしとは前提が違う。

代償は 47MB のモデルを古い端末に落として常時 WASM で回すこと。
ブラウザ実測ではフレーム平均 8.3ms / ヒープ 29MB。**古い実機では未測定。**

```bash
pnpm install
PORT=3020 pnpm dev   # 3020。nagara=3006 / admin=3002 と被らせない
pnpm exec tsc --noEmit
```

未導入: Supabase、Biome、PWA（Serwist）、GitHub リモート。

### 保存先（2026-07-17 決定）

原価をこちら側に溜めない、が方針。個人開発の規模で動画を自前で持つと回らない。

| 何を | どこに | なぜ |
| ---- | ------ | ---- |
| **動画** | **ユーザーの Google Drive** | 原価がゼロ。サービスが死んでもアルバムが残る |
| **画像**（SNS） | Supabase Storage | 軽い。tuzuri の sharp + WebP 変換パターンが効く |
| **日記・アカウント** | Supabase | |

- Drive の無料枠は **15GB で Gmail / Photos と共用**（[公式](https://support.google.com/drive/answer/6374270)）。
  溜まるほど価値が出る製品なので、容量で先に止まる可能性がある。課金のお願いは Google One に向く。
  ユーザー判断に委ねる方針（「ユーザー側にお好きにどうぞ」）。
- Drive API のレート制限は個人利用の規模では当たらない（1日 750GB まで）。効くのは容量だけ。
- ローカル保存は「ユーザーが Drive から自分で落とす」の意味。実装は不要。

### 配信

**最初は本人だけが見られる形にする。** 視聴者が1人ならコストはほぼゼロで、基盤の選択も後回しにできる。

**将来は「誰でも見れる」公開配信をやりたい**（ツイキャスのようにサムネが並び、押すと見られる）。
ただし帯域は動画と違って**ユーザー側に逃がせない**。人気が出るほど赤字が増える形になる。

#### 却下した案（蒸し返さない）

- **YouTube に配信させ、Tailee は一覧を出すだけ** — かわせない。YouTube Live の受け口は
  RTMP / RTMPS / HLS / DASH のみで、**外部向けの WebRTC 受け口が無い**
  （[公式](https://developers.google.com/youtube/v3/live/guides/ingestion-protocol-comparison)）。
  ブラウザから送るには自前サーバーで WebRTC → RTMP 変換が要り、全映像がそこを通る。
  入り口の帯域が残る上、見られなくても24時間ぶん払うので **Cloudflare Stream より高くつく**。
  加えてユーザーにチャンネル開設とライブ配信の有効化を要求することになる。
- **SkyWay**（nagara が使用）— 無料枠は**商用利用不可**、次が **Enterprise 月額11万円**から
  （[料金](https://skyway.ntt.com/ja/pricing/)）。ユーザーがゼロの日も出ていく固定費。
  nagara は施設向けの通話で事業が乗っているので条件が違う。

#### 有力（未決）

- **Cloudflare Stream** — $1 / 1,000分、**配信した分だけ**（[料金](https://developers.cloudflare.com/stream/pricing/)）。
  視聴者ゼロならタダ。入り口も向こう持ち。赤字が出るのは人が見に来ている時だけ。
  概算: 10人が毎日1時間で月18ドル、100人なら月2.7万円。回収の仕組みは未定。

### chappie

Tailee と同じ単一アプリ構成。`_components/` 規約、`ActionResult` を返す Server Actions、
`next.config.ts` のセキュリティヘッダ、Biome + lefthook + commitlint + knip + secretlint の周辺整備。

### nagara-kaigo（音声）

**ウェイクワード（Vosk）、AmiVoice のリアルタイム文字起こし、音声からの記録抽出は、あちらで一度通っている。**
同種の実装をする前に必ずあちらを読む。特に:

- `apps/nagara/src/stores/transcription-store.ts` — AmiVoice WebSocket
- `apps/nagara/src/stores/wake-word-store.ts` — ウェイクワード検知（実体は `wake-word-listener-vosk.tsx`）
- Vosk モデルは 47MB が下限（削減案は検討済みで全滅）
- ウェイクワードの誤発火は「よくある語」が原因になる。日常語を避ける
- `setSinkId({type:"none"})` の Bluetooth 回避と `audioContext.resume()` は忘れると Android で無音になる

### tuzuri（画像）

アップロード時に sharp で WebP 変換 → Supabase Storage。GIF は変換せず素通し。
パスを `{user.id}/{entityId}/{uuid}.webp` にして Storage の RLS を成立させる。
`apps/web/src/app/[locale]/(auth)/works/[workId]/actions.ts` を読む。
